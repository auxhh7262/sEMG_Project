## sEMG小程序 TCP连接 Bug 修复 (01:38 UTC / 09:38 CST)

### 根因分析
小程序配网成功后，realtime页面无法建立TCP连接。日志显示：
- `[BLE] ✅ 成功获取设备IP: 192.168.137.171 端口: 8888` ✅
- `[realtime] TCP未连接，尝试静默重连，cachedIp: 192.168.137.171` ✅
- **完全没有 `[TCP]` 开头的日志** → tcpClient.connect() 未被成功调用

### 发现的两个 Bug

**Bug 1: tcpClient.connect() 不返回 Promise**
- realtime/index.js 调用 `tcpClient.connect(...).then().catch()`
- 但 tcpClient.js 的 connect() 函数没有 return 语句，返回 undefined
- 导致 `.then()` 调用失败（TypeError），被静默吞掉

**Bug 2: 调用方式不规范**
- realtime/index.js 传入完整 URL: `tcpClient.connect('ws://${ip}:8888')`
- tcpClient.js 期望传入 IP 地址，内部自动构建 URL
- 虽然代码有清理逻辑（去 ws:// 前缀、去端口号），但用法不规范

### 已完成的修复

**tcpClient.js 修改：**
1. 顶部添加 `_connectResolve` 和 `_connectReject` 变量
2. connect() 函数改为返回 `new Promise((resolve, reject) => {...})`
3. onOpen 回调中 resolve Promise
4. onError/onClose 回调中 reject Promise
5. _cleanup() 中清理 Promise 引用

**realtime/index.js 修改：**
- `tcpClient.connect('ws://${ip}:8888')` → `tcpClient.connect(ip, 8888)`

## sEMG小程序 TCP连接 Bug 修复 (01:38 UTC / 09:38 CST)

### 根因分析
小程序配网成功后，realtime页面无法建立TCP连接。日志显示：
- `[BLE] ✅ 成功获取设备IP: 192.168.137.171 端口: 8888` ✅
- `[realtime] TCP未连接，尝试静默重连，cachedIp: 192.168.137.171` ✅
- **完全没有 `[TCP]` 开头的日志** → tcpClient.connect() 未被成功调用

### 发现的两个 Bug

**Bug 1: tcpClient.connect() 不返回 Promise**
- realtime/index.js 调用 `tcpClient.connect(...).then().catch()`
- 但 tcpClient.js 的 connect() 函数没有 return 语句，返回 undefined
- 导致 `.then()` 调用失败（TypeError），被静默吞掉

**Bug 2: 调用方式不规范**
- realtime/index.js 传入完整 URL: `tcpClient.connect('ws://${ip}:8888')`
- tcpClient.js 期望传入 IP 地址，内部自动构建 URL
- 虽然代码有清理逻辑（去 ws:// 前缀、去端口号），但用法不规范

### 已完成的修复

**tcpClient.js 修改：**
1. 顶部添加 `_connectResolve` 和 `_connectReject` 变量
2. connect() 函数改为返回 `new Promise((resolve, reject) => {...})`
3. onOpen 回调中 resolve Promise
4. onError/onClose 回调中 reject Promise
5. _cleanup() 中清理 Promise 引用

**realtime/index.js 修改：**
- `tcpClient.connect('ws://${ip}:8888')` → `tcpClient.connect(ip, 8888)`

## TCP连接仍失败的真正根因 (01:49 UTC / 09:49 CST)

### 上一轮修复后TCP仍未连接

用户反馈刷新后TCP仍无法连接。代码审查发现**真正的根因**：

**Bug 3（根因）：`_tabVisible` 从未在 onShow() 中被设为 true**

完整问题链路：
1. `onShow()` → 调用 `_silentReconnect(cachedIp)` → `setTimeout(tryReconnect, 1000)`
2. 1秒后 `tryReconnect()` 执行 → 第一行检查 `if (!this._tabVisible || retry >= 5) return`
3. `_tabVisible` 初始值是 `false`，且 **`onShow()` 从未将其设为 `true`**
4. → 直接 return，TCP 连接从未被发起！

**之前修复的 Bug 1 和 Bug 2 是真实的但不是根因**——因为 `_tabVisible` 检查在更早的位置就短路了，`tcpClient.connect()` 根本没被调用到。

### 修复方案（待执行）
- `onShow()` 入口处添加 `this._tabVisible = true`
- `_resumeStream()` 中也确保设置 `this._tabVisible = true`

## TabBar高亮 + IP缓存修复（13:34 CST）

### 1. TabBar高亮问题（已修复）

**症状**：扫描二维码进入小程序后，无论在哪个tab页，底部"校准建库"一直高亮。切换到其他页面也保持不变。

**根因分析**：`_syncSelected()` 里有 `if (this.data.selected !== target)` 的保护条件，如果 selected 已经是正确的值就不再 setData。在微信小程序里，Component 的 data 更新可能存在延迟，导致高亮卡在初始值(0=calibrate)不动。

**修复**：移除"相同则跳过"保护，改为**每次都强制 setData**，确保高亮始终和实际页面同步。同时增强了日志，编译后可在控制台看到 `[TabBar] _syncSelected — 页面: /pages/index/index → tab索引: 3(网络配置)` 这样的输出。

**代码变更** (`custom-tab-bar/index.js`):
```javascript
_syncSelected() {
  // ... 移除 "if (this.data.selected !== target)" 保护
  // 无论是否相同都 force setData，确保高亮一定同步
  this.setData({ selected: target });
}

switchTab(e) {
  // ... 也是先 setData 再 switchTab
  this.setData({ selected: idx });
  wx.switchTab({ url });
}
```

### 2. IP缓存失效问题（已修复）

**症状**：重新扫描二维码进入小程序，显示"没有缓存信息，需要重新配网"，即使之前已配网成功。

问题是每次配网都写 `config_step`，所以几乎永远不是 undefined，导致显示中间状态。

**修复**：简化逻辑——**只要有 cachedIp，直接显示 SUCCESS 状态**。

## ADC定时器中断根因分析与修复尝试 (08:09 UTC / 16:09 CST)

### 根因分析完成

**问题定位：IRQManager::addTimerOverflow() 未启用 NVIC 中断**

通过分析 FspTimer 和 IRQManager 源码发现：

1. **addTimerOverflow() 源码分析**：
   ```cpp
   bool IRQManager::addTimerOverflow(TimerIrqCfg_t &cfg, Irq_f fnc) {
       // ... 设置 ELC 事件链接 ...
       cfg.base_cfg->cycle_end_ipl = TIMER_PRIORITY;
       cfg.base_cfg->cycle_end_irq = (IRQn_Type)last_interrupt_index;
       *(irq_ptr + last_interrupt_index) = (uint32_t)fnc;  // 设置 ISR
       last_interrupt_index++;
       // ⚠️ 没有调用 R_BSP_IrqEnable() 启用 NVIC 中断！
   }
   ```

2. **对比 addGenericInterrupt()**：
   ```cpp
   bool IRQManager::addGenericInterrupt(GenericIrqCfg_t &cfg, Irq_f fnc) {
       // ... 配置 ...
       R_BSP_IrqEnable((IRQn_Type)last_interrupt_index);  // ✅ 启用 NVIC 中断
   }
   ```

3. **对比 UART 配置**：
   - addPeripheral(IRQ_SCI_UART) 中也显式调用 `R_BSP_IrqEnable()`

**结论**：Timer 的溢出中断在 IRQ vector table 中注册了 ISR，但从未在 NVIC 中启用，所以中断永远不会触发。
