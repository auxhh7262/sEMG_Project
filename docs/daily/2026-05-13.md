- 实际采样不是真正的ISR——timer中断只设 flag，loop 中才读取ADC+推入buffer
- 这意味着采样精度取决于 loop 执行速度，1kHz可能不稳定

- `handleSaveRecord()` 用 `new uint16_t[seqLen]` 动态分配两个数组
- RA4M1 堆空间有限，碎片化风险
- 建议用静态缓冲区或栈分配（seqLen ≤ 3000, 每个数组6KB）

### 🟡 StateMachine: getStateName() 缺少 ST_DB_FEATURE
- `_validTransition()` 中没有 ST_DB_FEATURE 的转换规则
- `getStateName()` 也没有 ST_DB_FEATURE 分支，会返回 "?"
- 但 AppController 直接用 `_stateMgr->transitionTo(ST_DB_FEATURE)` 绕过了验证

### 🟡 CalibrationManager: 截尾均值只按RMS排序
- 按 RMS 排序后取中间 80% 的 RMS 平均值是合理的
- 但 MDF 平均值也用 RMS 排序后的截尾区间，不是按 MDF 自身排序

- 实际影响有限：RMS异常通常伴随MDF异常

### 🟢 StorageManager: g_workBuf 互斥
- 全局4KB工作缓冲被A区镜像、A区写、CRC计算共享
- 当前通过"使用时占用，完成后释放"的约定管理，无运行时互斥锁
- 单线程环境下安全，但如果未来引入中断驱动的存储操作需注意

## 固件串口日志（11:06）

- `findMedianFrequency()` 持续返回 -1.0f → getMDF() 返回 0.0f

## 根因分析（关键）

### 现象
- REST阶段FFT正常，MDF=50-68Hz
- MAX阶段第一帧起MDF即为0，RMS初期仍3.77（≈REST水平）
- 不是信号幅度问题，而是FFT功率谱计算出错

### 可能根因1：1e-6阈值过低
- REST总功率约1.56e-6（≈阈值边界）
- MAX时DC偏移放大9x（baseline移位），低频bins积累大量功率
- bin1（~3.9Hz）贡献: `(sample_rate/4*N*offset²) / N = offset² * sample_rate/4`
  - MAX offset≈10mV → bin1功率≈0.0025（>>其他bins）
- bin1~bin2（<10Hz范围）积累大量功率，导致半功率阈值在低频bins已达标
- 正确MDF在10-250Hz范围外，被遗漏，返回-1.0f

### 可能根因2：FFT twiddle因子初始化
- twiddle factors在构造函数中预计算一次
- 可能因某种原因（数组未初始化、编译器优化）全部为0
- FFT输出变垃圾，power_spectrum全0或极小 → findMedianFrequency返回-1.0f

### 可能根因3：Snapshot时机问题
- `takeSnapshotIfNeeded()` 条件：`m_sampleIndex % m_fftWindowSize == m_snapshotIndex`
- MAX开始时snapshot恰好捕获REST→MAX边界数据
- FFT窗口包含混合信号（前半REST样本+后半MAX样本）
- DC跳变导致频谱泄露 → bin1功率占主导 → MDF计算失败

# 2026-05-13 (12:10) — MAX相 MDF=0.0 Bug 根因确认与修复方案

## 串口日志分析结果（11:06）

固件已烧录 v3.9.7 修复（DC移除、返回-1.0f、resetEMA），但 **MAX相 MDF 仍为 0.0 Hz**。

### REST 阶段 ✅
- RMS avg=3.52 mV, MDF avg=65.1 Hz
- `findMedianFrequency()` 正常返回 50-68 Hz
- 日志：`[CTRL] CALIB_REST: rms= 3.681, mdf= 63.7`

### MAX 阶段 ❌
- RMS avg=32.13 mV, **MDF avg=0.0 Hz**
- `findMedianFrequency()` 持续返回 -1.0f
- 日志：`[CTRL] CALIB_MAX: rms= 3.768, mdf= 0.0`（首帧即0）

## 根因定位

### 核心问题：功率分布异常导致 MDF 搜索失败

**现象：**
- REST：MDF=50-68 Hz（正常）
- MAX：MDF=0.0 Hz（全程，RMS 已从 3.7 涨到 54.4 mV）

**分析：**
1. `findMedianFrequency()` 中循环 `for (uint16_t i=1; i<num_bins; i++)` 遍历 bins
2. 只累加 `min_freq(10Hz)` 到 `max_freq(250Hz)` 范围内的功率
3. **MAX 阶段 DC 偏移放大 9 倍** → bin 1(~3.9Hz) 和 bin 2(~7.8Hz) 功率暴涨
4. 这两个 bin **在 10Hz 范围外**，不参与 `accumulated` 累加
5. 但它们消耗了大量总功率 → `half_power = total_power * 0.5` 偏小
6. 10-250Hz 范围内的累积功率在遍历完 bin 63 前 **永远达不到** `half_power`
7. → 循环结束未触发 `if (accumulated >= half_power)` → **返回 -1.0f**

**验证：**
- REST 阶段 DC 偏移小，bin 1-2 功率占比低 → `half_power` 相对大 → 10-250Hz 范围内能达标 → 返回有效 MDF
- MAX 阶段 DC 偏移大，bin 1-2 功率占比高 → `half_power` 相对小 → 10-250Hz 范围内达不到 → 返回 -1.0f

## 修复方案（4 步）

### 1. 跳过 bin 1-2（0-10 Hz 生理噪声范围）
```cpp
for (uint16_t i = 3; i < num_bins; i++)  // 原 i=1，改 i=3
```

### 2. 初始化 `m_currentMDF = 50.0f`
在 `SignalProcessor` 构造函数中：
```cpp
m_currentMDF = 50.0f;  // 原 0.0f
m_lastValidMDF = 50.0f;
```

### 3. MDF 无效时 fallback 到上次有效值
在 `calculateMDF()` 中，当 `m_rawMDF < 0.0f` 时：
```cpp
if (m_rawMDF < 0.0f) {
    // FFT 无效，返回上次有效 MDF，不修改状态
    return m_currentMDF;  // 原 return m_currentMDF（但 m_currentMDF 可能=0.0）
}
```

### 4. 降低 `total_power` 阈值
```cpp
if (total_power < 1e-12f) return -1.0f;  // 原 1e-6f
```

| 校准 | REST MDF | MAX MDF | 结果 |

## 根因链
1. `resetEMA()` 设 `m_lastValidMDF = 80.0f`
2. 信号弱 → `findMedianFrequency()` 返回 `m_lastValidMDF=80.0f` 作为 fallback
3. 80.0f 在 [10,250]Hz 范围内 → 被当真实 rawMDF → **永远锁死**

## 修复方案C（v3.9.7已编码09:27）
- findMedianFrequency() 异常时返回 -1.0f（不返回 m_lastValidMDF）
- calculateMDF() 检测 -1.0f 时跳过EMA更新
- resetEMA() m_lastValidMDF 初始值 80.0f → 0.0f
