## 校准完整分析结论 (~09:33-09:35 CST)
- **固件逻辑完全正确，无需代码修改**
- WebSocket 校准测试（_tw_calib_full2.py）结果：
  - 207条数据在 28s 内收到（rms≈1410-1460, mdf≈36.2）
  - fatigue=0, act=0 始终不变
  - +28s 后数据流中断 3s（对应 REST 10s + MAX 15s + 处理时间）
- **根因：传感器悬空/开路**
  - 悬空时 RMS ≈ 1450mV（正常人体应 <100mV），MDF 固定 36.2Hz（正常 70-120Hz）
  - 校准 REST ≈ MAX（相同信号），验证规则2失败：ref_rms ≤ rest_rms * 2
  - 即使通过，规则5也会失败：ref_mdf ≈ rest_mdf（MDF 未下降 ≥10%）
  - 校准验证失败 → 进入 ERROR 状态 → 停止数据推送 → 28s 后无数据
- **fatigue/act=0 的原因**：
  - activation = (rms - restRMS) / (maxRMS - restRMS)，悬空时分母≈0
  - rms > restRMS * 2.0 判断收缩，悬空时不满足 → 不计算疲劳
- **验证方法**：需要将传感器正确贴在人体肌肉上测试
- **补充发现**：串口 raw TCP 连接无法触发 WebSocket 命令处理（8888 端口是 WebSocket 协议，非 raw TCP）
  - 之前用 serial_calib.py 发 raw TCP 数据，固件不响应（[PROTO] 日志不存在是正常的）
  - WebSocket 测试（_tw_calib_full2.py）才是正确的通信方式
- 测试脚本：_tw_boot_monitor.py, _tw_serial_calib.py, _tw_calib_full2.py, _tw_build_upload.py

## 小程序校准流程修复 (~10:13 CST)
**问题**：校准进度卡在24%，MAX阶段倒计时缺失
**根因**：`startCalibration()`过早发送`{"cmd":"start"}`，固件立即进入REST并推送数据，但小程序10秒倒计时期间未注册监听器，数据全部丢失
**修复方案**：
1. `startCalibration()`移除`tcpClient.send('{"cmd":"start"}')`和`this._registerTcpListener()`
2. REST倒计时结束后由`_beginCollecting('resting')`统一发送start命令+注册监听器
3. `_beginCollecting(phase, sendStart=true)`新增参数，MAX阶段传false（固件自动切换）
4. `_beginCollecting`中增加MAX阶段15秒UI倒计时
5. `_finishPhase('resting')`调用`_beginCollecting('max_contraction', false)`
6. 同样修复`_startDbCalibration`和`_beginDbCalibCollecting`（建库流程）
7. WXML countdown条件增加`phase === 'max_contraction'`

**修改文件**：
- `pages/calibrate/index.js`：startCalibration, _beginCollecting, _finishPhase, _startDbCalibration, _beginDbCalibCollecting, _finishDbCalibPhase
- `pages/calibrate/index.wxml`：countdown显示条件

## 小程序校准进度卡 73% 问题 (~10:21 CST)
**用户报告**：点击校准→倒计时→REST/MAX阶段→MAX倒计时结束后进度73%→"采集进行中"卡住
**根因**：样本采集目标与固件实际推送速率不匹配
- 固件推送 ~8Hz（非代码注释假设的10Hz）
- REST: 固件10s×8Hz=80样本，但COLLECT_SAMPLES=100 → REST延迟完成，"偷走"MAX时间
- MAX: 固件15s×8Hz=120样本，但MAX_COLLECT_SAMPLES=150 → 永远达不到目标
- 实际MAX可得样本 ≈ (15 - 偷走时间) × 8 ≈ 100 → 100/150 ≈ 67%（与用户报告73%吻合）

**修复方案**：在`_beginCollecting`和`_beginDbCalibCollecting`中增加时间兜底机制
- REST: 11秒后强制完成（固件10s + 1s缓冲），最少30样本
- MAX: 16秒后强制完成（固件15s + 1s缓冲），最少30样本
- `_finishPhase`/`_finishDbCalibPhase`增加`_phaseCompleted`防重入保护
- `_cleanup`增加`_phaseTimer`清理
- 样本数达标仍作为快速完成路径保留

**MDF不变问题**：仍然是电极接触不良（MDF卡在20-24Hz，正常70-120Hz），非代码问题

**待测试**：用户刷新小程序后重新校准（需电极贴紧皮肤）
