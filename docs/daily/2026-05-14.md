## v3.9.10 — RMS 偏低 50 倍根因找到！🔧

### 根因
`takeSnapshotIfNeeded()` 对 `m_snapshot[]` **原地排序**来算裁剪均值 DC 偏移
- 排序后数组失去时序 → RMS 计算中 AC 分量被严重压缩 → RMS 极低（6-8mV vs 正常 300mV）
- FFT/MDF 碰巧仍正常（排序后频谱结构未完全破坏）
- 这是之前"优化"引入的 bug

### 修复
去掉原地排序，改用简单均值算 DC 偏移（512样本均值已足够稳定），保持快照时序完整性
- 编译：RAM 69.8%, Flash 57.8%
- 已上传，等待校准验证

### 教训
⚠️ **共享快照缓存不能原地排序！** 修改共享缓冲区时必须保证后续消费者（RMS/FFT）的数据完整性

## 算法问题分析（10:34-10:38）

### 四个算法问题已识别

**1. 激活度(act)永远100%** 🔴 最严重
- calib_max_rms=6.44mV < monitoring_rms=11.26mV
- act公式溢出被clamp到1.0，完全失去意义
- 根因：MAX阶段RMS从2mV渐变到11mV，截尾均值只得到6.44mV（既非静息也非最大收缩）

**2. 截尾均值对渐变信号无效**
- 10%截尾均值假设离群点是孤立的，但实际是信号从弱到强渐变
- MAX阶段前5个样本≈2mV（未真正收缩），后7个≈9-11mV（真正收缩）
- 截掉10%后剩下80%仍混合了非收缩和收缩数据

**3. REST阶段数据不稳定**
- MDF CV=95.1%（147→91Hz暴跌），EMA残留+传感器接触在变化
- SNAP_DBG显示REST初期峰峰值34counts→后期66counts
- 前3-4秒数据不可靠

**4. 疲劳度公式对短期收缩不适用**
- fatigue = 1 - (current_mdf / rest_mdf)
- 收缩时MDF先升后降（recruitment→fatigue），current_mdf > rest_mdf → 负值 → 截断0
- 只适用于长时间持续收缩MDF跌破基线的场景

### 建议修复方向
1. MAX阶段改用后半段均值或峰值RMS，不用全阶段截尾均值
2. 上升沿检测：找到RMS显著上升的时间点，只用之后的数据
3. REST丢弃前几秒（EMA残留+接触稳定期）
4. 疲劳度用MDF下降速率或与initial_contraction_mdf比较

### 优先级
先修MAX阶段RMS计算方式（直接导致act失效）

2. **REST极不稳定**：CV>45%说明接触从"几乎悬空"到"轻微接触"在变化
3. **MAX RMS CV=141.6%**：数据极不均匀（2mV跳到11mV）
4. **根因：干电极接触质量差**，皮肤阻抗高导致信号衰减10-20倍
5. ADC参考5V确认正确（DCbias≈1.59V≈传感器基准1.5V）
6. 固件算法无问题，RMS/MDF计算逻辑正确

## 算法问题分析（10:34-10:38）

### 四个算法问题已识别

**1. 激活度(act)永远100%** 🔴 最严重
- calib_max_rms=6.44mV < monitoring_rms=11.26mV
- act公式溢出被clamp到1.0，完全失去意义
- 根因：MAX阶段RMS从2mV渐变到11mV，截尾均值只得到6.44mV（既非静息也非最大收缩）

**2. 截尾均值对渐变信号无效**
- 10%截尾均值假设离群点是孤立的，但实际是信号从弱到强渐变
- MAX阶段前5个样本≈2mV（未真正收缩），后7个≈9-11mV（真正收缩）
- 截掉10%后剩下80%仍混合了非收缩和收缩数据

**3. REST阶段数据不稳定**
- MDF CV=95.1%（147→91Hz暴跌），EMA残留+传感器接触在变化
- SNAP_DBG显示REST初期峰峰值34counts→后期66counts
- 前3-4秒数据不可靠

**4. 疲劳度公式对短期收缩不适用**
- fatigue = 1 - (current_mdf / rest_mdf)
- 收缩时MDF先升后降（recruitment→fatigue），current_mdf > rest_mdf → 负值 → 截断0
- 只适用于长时间持续收缩MDF跌破基线的场景

## v3.9.10c — 疲劳/激活度算法重构（10:50-11:07）

### 疲劳度算法改为 MDF-only
```
fatigue = 1.0 - (current_mdf / calib_rest_mdf)
```
- 放弃 RMS 参与疲劳计算（收缩力度干扰方向不确定）
- 仅用 MDF 下降比例（MDF 随疲劳下降，生理学依据充分）

### 激活度算法改为 MDF-only
```
activation = 1.0 - (current_mdf / calib_max_mdf)
```
- 肌肉激活时 MDF 先升后降，高激活 → MDF 升高
- 与 fatigue 对称，逻辑一致

### 收缩起点追踪（m_contractionStartMDF）
- `_handleMonitoringState()` 中，当 RMS 超过 `rest_rms + 0.3*(max_rms-rest_rms)` 时记录初始 MDF
- `updateFatigue()` 比较 `m_contractionStartMDF` 与当前 MDF

## 小程序 UI 改动（realtime/index 页面 — 12:55 完全重设计 + 迭代优化）

### 新布局结构（最终版）
**上半部分 — 数据区 `.data-section`**：
- 第1行：标题栏 `.dt-header`（列名+单位，如 "RMS (mV)"、"MDF (Hz)"）
- 下面最多5行：实时数据帧（`.dt-row`），旧帧在上、最新帧在下
- 最新帧最后一行加粗高亮（`.dt-row-latest`：深色背景+大字）

**下半部分 — 指标说明 `.legend-section`**：
- 靠左对齐，字体加大（标签28rpx、说明26rpx）

### 13:03 修复 — 数据行不显示问题
- **根因**：`.data-section` 设了 `flex-shrink: 0` 导致不伸缩；`scroll-view` 在 flex 布局中高度计算为0
- **修复**：
  - `.data-section`: `flex-shrink: 0` → `flex: 1`（占据上半部分空间）
  - WXML: `<scroll-view>` → 普通 `<view class="dt-body">`（避免 scroll-view 高度 bug）
  - `.dt-body`: `flex: 1; overflow-y: auto; min-height: 0`

### 微信小程序 scroll-view 经验教训
- ⚠️ scroll-view 在 flex 容器中经常高度塌陷，必须给固定 height 才能正常工作
- 数据量少（<10条）时用普通 view + overflow-y:auto 更可靠

### 显示顺序
- historyRows 用 push（新帧追加末尾），旧→新从上到下
- 高亮条件：`index === historyRows.length - 1`（最后一行最新帧）

### JS 变化
- MAX_HISTORY = 5
- 删除 rmsStr/actStr/mdfStr/fatStr/timeStr 独立字段，统一走 historyRows
- 最新帧也 push 进 historyRows

### 其他页面改动
- 校准页：REST 10秒倒计时（与MAX一致），直接开始采集
- 时间戳：new Date(ts) 而非 ts*1000
