3. SSID写入触发 `onSsidWritten()` → `_newSsid`
4. Password写入触发 `onPassWritten()` → `_hasNewCredentials = true` + `BLE.stopAdvertise()`
5. `NetManager::_tryConnectWifi()` 消费凭证：`hasNewCredentials()` → `consumeCredentials()`
6. `WiFi.disconnect(true)` 断开旧连接 → delay(200) → `WiFi.begin(ssid, pass)`
7. `gStorage.SaveWifiCredentials()` 保存到EEPROM
8. `NetManager::tick()` 检测 `WiFi.status() == WL_CONNECTED` → `_wifiConnected = true`
9. `BleConfigServer.tick()` 检测WiFi连接 → `writeLocalIpCharacteristic()` 写入IP
10. BLE重连后 `restartAdvertising()`

**TCP服务器启动条件（重要发现）**：
- TCP服务器在 `NetManager.init()` 中无条件启动（`_tcpServer.begin()`）
- 不依赖WiFi连接状态 → 设备开机后即可通过TCP连接

### 🔑 关键发现：COM4 被 wechatdevtools 占用

进程列表中明确看到：
```
PID 9344  wechatdevtools  真机调试
```

微信开发者工具的「真机调试」模式（Real Device Debugging）通过 USB/ADB 连接物理手机，会占用设备的 COM 端口（Arduino UNO R4 作为 CDC-ACM 设备）。这导致：
- `pio sam-ba`（bossac 烧录工具）→ Permission denied
- 串口监视器 → 无法打开 COM4

### 解决步骤
1. **关闭微信开发者工具的「真机调试」模式**，或直接退出微信开发者工具
2. 等待 5 秒
3. 重试 `pio run -t upload`
4. 串口监视器也可以正常启动了

### 🔑 核心发现：bossac 需要设备处于 bootloader 模式

bossac 的工作流程：
1. 通过 1200bps + DTR/RTS 触发重启 → 设备进入 bootloader 模式
2. bootloader 在 8 秒内等待 bossac 连接
3. 连接后进行 flash 读写

**问题根源**：UNO R4 可能处于"正常运行模式"（固件正在运行），而不是 bootloader 模式
- 在正常运行模式下，CDC-ACM 串口正常工作（COM4 可见），但 bossac 无法通信
- bossac 需要 bootloader 的协议，而固件运行的 CDC-ACM 不支持 bossac 协议

- 方法2：在 PowerShell 中用 1200bps + DTR/RTS 打开 COM4 触发重启（但这需要独占访问，而 COM4 被占用）
- **推荐：双击复位按钮**

## 2026-05-09 16:19 内存压缩flush（NTP时序bug修复）

### Bug：NTP在DHCP完成前发送，必然失败

用户串口日志明确显示：
```
[WIFI] Auto-connecting from EEPROM: LT02
[NTP] Requesting time sync...        ← IP还是0.0.0.0，UDP包发不出去
[WIFI] Connected! IP: 192.168.137.171  ← DHCP完成后才拿到IP
[NTP] Sync timeout, will retry in 60s
```

**根因**：`NetManager::tick()` 中，`WiFi.status() == WL_CONNECTED` 后立即调用 `syncTime()`，但此时 DHCP 还没完成，IP 仍为 0.0.0.0，NTP UDP 包无法路由。

- **修改文件**：NetManager.cpp
- **修改位置**：`tick()` 中 WiFi 连接成功后的处理块
- **修改前**：`_wifiConnected = true` → `syncTime()` → DHCP 等待循环 → BLE IP 通知
- **修改后**：`_wifiConnected = true` → DHCP 等待循环 → BLE IP 通知 → `syncTime()`（仅 DHCP 成功时）
- 如果 DHCP 超时，不调 NTP，等 60 秒后重试机制触发

## 2026-05-09 16:29 内存压缩flush（NTP根因修复 + DNS错误处理 + 启动分隔线）

### 🔑 关键Bug修复：NTP永远失败的根因 — `_ntpUdp.begin()` 从未调用

WiFiUDP 对象必须先调 `begin(localPort)` 分配本地端口，才能收发 UDP 包。
之前代码直接调 `beginPacket()`/`write()`/`endPacket()`，等于往一个没打开的端口发包 → 发不出去也收不回来 → 永远超时。

**修复**：在 `NetManager::init()` 中添加 `_ntpUdp.begin(8889)`

### DNS错误处理（syncTime增强）

`beginPacket(hostname, port)` 内部会做 DNS 解析，但如果 DNS 失败（路由器不支持/网络未就绪），之前静默失败无任何日志。

**修复**：先手动调 `WiFi.hostByName()` 解析 DNS，失败则打印日志并 60 秒后重试。
修复后日志示例：
```
[NTP] Requesting time sync from ntp.aliyun.com (203.107.6.88)...
[NTP] Sync successful. Base time: 1746787320
```

### 启动分隔线（防混淆）

开机日志前两行 `[STORAGE] WiFi creds loaded` 和 `[WIFI] Auto-connecting` 是**上次启动的串口缓冲区残留**，不是本次输出（gStorage.Init() 在 gNetManager.init() 之前调用）。

**修复**：在 `setup()` 开头加了 `========== sEMG V1.4 BOOT ==========` 分隔线

## 2026-05-09 17:35 内存压缩flush（NetManager BLE配网修复）

### 🔑 关键修复：设备已连WiFi时无法重新配网

**根因**：`NetManager::tick()` 中，`_tryConnectWifi()` 仅在 `!_wifiConnected` 时被调用，而 BLE 凭证消费逻辑在 `_tryConnectWifi()` 内部 → 当设备已连接WiFi时，新BLE凭证被完全忽略。

**修复方案**：在 `tick()` 开头添加最高优先级BLE凭证检测：

```cpp
void NetManager::tick() {
    // [BLE-Config-Fix] 最高优先级：检测 BLE 新凭证
    if (_bleServer && _bleServer->hasNewCredentials()) {
        WifiCredentials_t creds = _bleServer->consumeCredentials();
        LOG("[WIFI] New BLE credentials received, reconnecting to: %s\n", creds.ssid);
        
        WiFi.disconnect();
        delay(200);
        gStorage.SaveWifiCredentials(&creds);
        WiFi.begin(creds.ssid, creds.pass);
        
        _wifiConnected = false;
        _eepromCredsTried = true;
        _wifiRetryTimer = millis();
        return;
    }
    
    // ... 原有逻辑
}
```
